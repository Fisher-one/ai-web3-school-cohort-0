|------|------|
| **名称** | Chain Research Agent |

## 二、能力声明（Capability Manifest）

### 可执行能力

| 能力 | 具体动作 | 是否需要链上操作 |
|------|---------|----------------|
| 查询链上公开数据 | 读取 Uniswap / Curve 事件日志，解析合约状态 | 只读，免费 |
| 调用付费数据 API | 通过 x402 支付 Dune Analytics / The Graph 等服务 | 需要 ERC-20 transfer |
| 生成分析报告 | 整合多源数据，输出 Markdown 格式分析结果 | 不需要 |
| 验证数据真实性 | 对 API 返回数据做签名验证和 hash 记录 | 不需要 |
| 执行预算内转账 | ERC-20 transfer / approve，单笔 ≤ 10 USDC，日限 50 USDC | 需要，在 Session Key 范围内自动执行 |

### 明确禁止的动作

- 合约部署 / 升级
- 治理投票
- 超出白名单合约的任何调用
- 私钥或助记词操作
- 修改 Session Key 规则本身

## 三、输入与输出

### 输入

```
用户自然语言需求（示例）：
"查一下过去 7 天 Uniswap 上 USDC/ETH 池的交易量变化，对比 Curve 同池数据，给我一份报告。"
```

附带上下文：
- 用户预设的预算上限（本次任务 ≤ 10 USDC）
- 白名单数据服务商列表（Dune、The Graph 等）
- Session Key 授权范围

### 输出

```
1. 数据分析报告（Markdown）
   - 数据来源 + 获取时间
   - 关键指标对比（交易量 / 手续费 / 流动性）
   - 结论与异常点标注

2. 执行日志
   - 任务 ID、时间戳
   - 支付记录（tx hash）
   - 数据 hash + 服务商签名
   - 总花费
```

## 四、协作对象

```
用户
  │ 下达任务 + 设定预算
  ↓
Chain Research Agent
  ├─── 链上数据（直接读取，免费）
  │     Uniswap / Curve 合约事件
  │
  ├─── 付费数据服务（x402 支付）
  │     Dune Analytics / The Graph
  │
  ├─── 智能账户 / Session Key（执行支付）
  │     ERC-4337 账户 + Policy 验证
  │
  └─── 用户（最终验收）
        报告质量判断，不可自动化
```

## 五、如何被发现和调用

**当前阶段（v0.1）**：用户本地直接调用，无需发现机制。

**未来扩展（如接入 Agent Registry）**：

Agent 可以发布一个机器可读的 Capability Manifest，让其他 Agent 或用户通过 Registry 发现它：

```json
{
  "agent_id": "chain-research-agent",
  "owner": "0xFisher...",
  "capabilities": ["onchain_query", "paid_api_call", "report_generation"],
  "budget_limit": "50 USDC/day",
  "supported_protocols": ["x402", "ERC-4337"],
  "verification": "ERC-8004",
  "endpoint": "https://agent.fisher.eth/invoke"
}
```

调用方式：POST 请求 + 任务描述 + 预算授权。

### Profile 如何更新

**当前（v0.1）**：用户本地修改配置文件，Agent 启动时重新加载。只有用户本人可以更新——Agent 不能自己修改自己的能力声明。

**未来（接入 Registry 后）**：通过链上交易更新 ERC-8004 声明。旧版本保留历史记录，新版本生效后旧版本自动失效。任何调用方都可以查到「这个 Agent 当前声明的能力是什么版本、什么时候更新的」。

## 六、如何被验证

### 身份验证（ERC-8004）

链上声明「这个 Agent 由 0xFisher... 授权」。其他服务方（如 Dune）收到付款时，可以验证：
- 付款来自授权 Agent（不是随机地址）
- 授权在有效期内
- Agent 的能力范围包含「购买数据」

### 执行记录验证

每次任务完成后，以下内容可供事后审计：
- 链上 tx hash（支付记录，不可篡改）
- 数据 hash + 服务商签名（证明数据来源和完整性）
- 任务日志 hash（证明 Agent 声称的执行过程）

### 当前局限

**报告质量没有链上可验证性**。「分析结论准不准」这件事，目前只有人工验收一条路，没有自动验证机制。这是这个 Agent 信誉体系里最大的空白。

## 七、失败处理

| 失败类型 | 触发条件 | 处理方式 |
|---------|---------|---------|
| Policy 拒绝 | 金额超限 / 地址不在白名单 | 记录原因 + 通知用户，不重试 |
| 数据签名验证失败 | 服务商返回数据被篡改 | 拒绝使用该数据，通知用户，发起退款请求（链下） |
| 链上交易失败 | Gas 不足 / 网络拥堵 | 记录 + 通知，等用户决定是否重试 |
| 服务商无响应 | 付款成功但数据未返回 | 记录异常 + 通知用户，链下投诉（无自动仲裁） |
| 报告质量不达标 | 用户拒绝验收 | Agent 用已购数据重新生成，或用户放弃（损失数据费） |

## 八、当前版本的边界

这个 Agent 设计回答了三个问题，也留下了三个空白：

**已回答：**
- 在授权范围内能做什么（权限策略）
- 怎么花钱（x402 支付流程）
- 失败了怎么处理

**还没回答：**
- **报告质量怎么客观衡量**：目前只有人工验收，没有自动评估机制
- **信誉怎么积累**：Agent 做了 100 次好报告，这个历史记录存在哪、谁来认证
- **付了钱服务不交付怎么办**：x402 没有 Escrow，Dune 跑路了无法自动退款

## 九、可选对比：MCP vs A2A

### MCP（Model Context Protocol）：Agent 和工具之间的接口

MCP 解决的是「Agent 怎么调用工具」——定义了工具的输入输出格式、上下文传递方式。

在本场景中：Agent 调用 Dune API、读取链上事件——这些工具调用的接口规范就是 MCP 要解决的问题。有了标准接口，Agent 不需要为每个工具单独写适配层。

**适合的场景**：单个 Agent 调用多种工具（数据库、API、合约读取）时，需要统一的上下文和调用格式。

### A2A（Agent to Agent Protocol）：Agent 之间的协作接口

A2A 解决的是「两个 Agent 之间怎么交换任务和结果」——当 Agent A 需要雇用 Agent B 完成一个子任务时，双方怎么沟通任务描述、进度、结果和报酬。

假设未来 Chain Research Agent 需要雇一个专门的「图表生成 Agent」来可视化数据——这两个 Agent 之间的协作协议就是 A2A 的场景。

**适合的场景**：多 Agent 协作，一个 Agent 作为 orchestrator 把子任务分发给其他专业 Agent。

### 两者的关键区别

| 维度 | MCP | A2A |
|------|-----|-----|
| 协作对象 | Agent ↔ 工具 | Agent ↔ Agent |
| 解决的问题 | 工具调用接口标准化 | 任务委派和结果交换 |
| 是否涉及支付 | 不直接涉及 | 可以结合 MPP 实现 Agent 间结算 |
| 本场景中的角色 | Agent 调用 Dune API 时用 | 未来多 Agent 协作时用 |
