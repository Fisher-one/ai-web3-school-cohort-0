## 一、执行流程图：哪些步骤可以自动化，哪些必须人工确认

```
用户设置 Pact（一次性人工确认）
    ↓
[用户签名] 配置 Session Key 规则：
    - 预算上限：每日 ≤ 50 USDC
    - 可调用合约白名单
    - 每日最多 5 笔
    - 有效期：30 天
    ↓
━━━━━━━━━━━━━━━━ 以下进入自动执行循环 ━━━━━━━━━━━━━━━━
    ↓
Agent 发起意图（UserOperation）
    ↓
Policy 检查（自动）
    ├─ 金额是否超出单笔上限？
    ├─ 今日笔数是否超出频率限制？
    ├─ 收款地址是否在白名单内？
    └─ Session Key 是否过期？
    ↓
         ┌── 任意一项不通过 ──→ ❌ 自动拒绝 + 告警通知用户
         ↓ 全部通过
Guard 模拟执行（Simulation）（自动）
    ├─ 余额变化是否符合预期？
    ├─ 是否存在 re-entrancy：目标合约是否在回调中重新进入？
    ├─ 是否有 delegatecall 到未知地址（可能改写 Storage）？
    ├─ 是否触发 selfdestruct（直接销毁合约）？
    ├─ 是否有大额 transfer 到非白名单地址（即便 Policy 通过仍做二次校验）？
    └─ Gas 估算是否合理？
    ↓
         ┌── 模拟结果异常 ──→ 🔴 暂停，等待人工确认
         ↓ 模拟结果正常
风险分级（自动）
    ├─ 低风险（金额 ≤ 10 USDC，已知合约）──→ ✅ 自动执行
    └─ 高风险（金额 > 10 USDC，或首次调用合约）──→ 🔴 人工确认
    ↓
🔴 人工确认节点（高风险时触发）
    用户看到：操作摘要 / 金额 / 收款方 / 预估 Gas
    用户选择：确认 / 拒绝 / 修改参数
    ↓
链上广播（EntryPoint 合约验证 + 执行）（自动）
    ↓
结果验证（自动）
    ├─ tx hash 确认
    ├─ 链上状态与预期一致？
    └─ 写入日志（时间戳 / 金额 / 地址 / tx hash）
    ↓
         ┌── 执行失败 ──→ 失败处理流程（见下）
         ↓ 执行成功
Agent 通知用户结果 ✅
```

### 具体场景走一遍

**场景 A：正常低风险任务**

> Agent 收到任务：「给 DataDAO 续费订阅，金额 8 USDC」

- Policy 检查：8 < 10 ✓，地址在白名单 ✓，今日第 2 笔（< 5）✓，Session Key 未过期 ✓
- Guard Simulation：余额从 120 → 112 USDC，符合预期 ✓，无异常事件 ✓
- 风险分级：低风险（已知合约 + 8U）→ 自动执行
- 结果验证：链上状态一致 ✓
- 通知用户：「已自动完成 DataDAO 订阅续费，tx: 0xabc...」

全程无需用户介入。

**场景 B：高风险任务触发人工确认**

> Agent 收到任务：「给一个上周才出现的 DeFi 协议转 45 USDC」

- Policy 检查：45 > 10 ✗，且合约不在白名单 ✗
- 立即挂起，推送人工确认：「待执行操作：转 45 USDC 至未知合约 0xXYZ...（7 天前创建）。该操作超出自动执行阈值，需要您确认。」
- 用户看到摘要后选择：拒绝。
- 日志记录：挂起原因 + 用户拒绝 + 时间戳。

---

### 自动 vs 人工确认汇总

| 步骤 | 执行方 | 原因 |
|------|--------|------|
| Policy 规则检查 | 自动 | 确定性规则，无需判断 |
| Guard Simulation 预演 | 自动 | 机器执行，结果可验证 |
| 低风险交易执行（≤10U，已知合约） | 自动 | 在 Pact 授权范围内 |
| **首次调用新合约** | **🔴 人工确认** | 未知合约风险不可预估 |
| **单笔 > 10 USDC** | **🔴 人工确认** | 超出低风险阈值 |
| **Simulation 结果异常** | **🔴 人工确认** | 模拟显示非预期行为 |
| **Session Key 配置（Pact 设置）** | **🔴 人工确认** | 规则本身必须由人定 |
| **紧急暂停 / 撤销** | **🔴 人工操作** | 涉及安全边界变更 |

## 二、权限策略设计

场景：Agent 帮用户管理日常小额链上支付（订阅服务、API 调用费用、数据购买）

### 完整权限策略表

| 维度 | 策略配置 | 设计理由 |
|------|---------|---------|
| **预算** | 单笔 ≤ 10 USDC；每日总额 ≤ 50 USDC | 双重限制，防止单笔大额和小额多笔两种攻击路径 |
| **可调用合约** | 白名单制，初始为空，用户逐一添加 | 未知合约 = 未知风险，默认拒绝比默认允许更安全 |
| **可执行动作** | 仅允许：ERC-20 transfer / approve（有限额）；禁止：合约部署、升级、治理投票、私钥操作 | 高风险动作一律不进自动化，必须人工处理 |
| **人工确认阈值** | 单笔 > 10 USDC；首次调用新合约；Simulation 异常；Session Key 续期 | 阈值之下无感知，阈值之上必须人确认——对应 Elytro 的「免密支付上限」设计 |
| **频率限制** | 每日 ≤ 5 笔；单小时 ≤ 2 笔 | 防止攻击者在上限内高频刷走余额 |
| **有效期** | Session Key 30 天自动过期 | 强制用户定期重新核实规则，避免长期信任过期配置 |
| **撤销方式** | 三种：① 30 天自动过期 ② 用户随时主动撤销 ③ Guard 检测到异常自动暂停并通知。撤销后 in-flight 交易采用「安全优先」策略：撤销立即生效，pending 中未上链的交易全部拒绝（见下方说明） | 撤销不能只靠用户主动，系统必须有自动触发机制 |
| **日志记录** | 每笔操作记录：时间戳 / 操作类型 / 金额 / 合约地址 / tx hash / 执行结果 / 触发规则 | 审计和事后追责的基础，出了问题能还原完整链路 |
| **失败处理** | 区分两类失败：① Policy 拒绝 → 记录原因 + 通知用户，不重试 ② 链上执行失败（Gas 不足等）→ 记录 + 通知，等用户决定是否重试 | 两类失败原因不同，处理方式不能混用。Policy 拒绝不重试，链上失败可重试 |

### 为什么白名单地址必须用户手动填

最直觉的想法是让 Agent 自动从官网抓取收款地址。但这里有个根本问题：

**如果 Agent 可以修改规则本身，那规则保护的是谁？**

- Agent 可能从被篡改的页面获取到错误地址
- Agent 可能受到 prompt injection 攻击，被诱导填入攻击者地址
- 白名单一旦写入，Agent 就会在这个地址上自动转账

结论：**规则由人定，执行才交给 Agent**。两件事不能颠倒。

### 撤销后 in-flight 交易的处理

撤销那一刻，可能有已签名但尚未上链的交易在 Bundler 队列里。这是一个真实的设计取舍：

| 方案 | 行为 | 适用场景 |
|------|------|---------|
| **安全优先（本设计采用）** | 撤销立即生效，pending 交易全部拒绝 | 安全事件触发的撤销——此时宁可损失一笔 Gas，也不能让可疑交易上链 |
| 用户体验优先 | 已签名的交易允许完成，仅拒绝后续新交易 | 用户主动「下班」撤销——最后一笔正常操作不应被中断 |

本设计选择安全优先，原因：触发自动撤销的场景通常是 Guard 检测到异常，这种情况下 pending 交易本身就是可疑的，不应放行。

## 三、ERC-4337 / Safe / Guard 解决的是哪类风险

### ERC-4337：给账户加一个「规则中间层」

**EOA 的根本问题**：签名 = 立刻执行，没有中间层。私钥到手，什么规则都绕过去了。

ERC-4337 把执行流程拆成两步：

```
用户意图（UserOperation）→ 验证层（Policy / Guard）→ 执行（EntryPoint 合约）
```

这个验证层是所有权限控制逻辑的家：Session Key、支出限额、地址白名单、时间窗口——全部住在这里。

**核心洞察**：没有 ERC-4337 的话，EOA 的签名就是最终裁决——你可以在链下做一万层 Policy 检查，攻击者拿到私钥后全部跳过。ERC-4337 把验证逻辑搬到了链上验证阶段（`validateUserOp`），跳过验证 = 交易压根不会上链。这是从「建议」到「强制」的质变。

**解决的风险**：私钥被盗后攻击者仍受规则约束（Session Key 范围之外的操作无法执行）；Agent 被攻击后只能在授权范围内造成损失。

### Safe：多签 + Guard 插槽

Safe 是 ERC-4337 之上的具体实现，核心设计是**交易执行前必须过 Guard 合约**：

```
提案 → m/n 签名通过 → Guard.checkTransaction() → 执行 → Guard.checkAfterExecution()
```

Guard 插槽让任何人都可以写自定义检查逻辑并插入执行链路，不需要改 Safe 核心合约。

**解决的风险**：团队/DAO 金库的多人授权问题；通过 Guard 实现「合约层面的确定性拦截」，不依赖 AI 判断。

### Guard / Policy：AI 判断层 和 链上执行层之间的翻译器

AI 的判断是概率性的——它可能被 prompt 注入影响，可能产生幻觉，可能误判风险。链上执行是不可逆的。

**两者之间必须有一层确定性规则**：

| 层 | 特点 | 做什么 |
|----|------|--------|
| AI 判断层 | 概率性，可解释，灵活 | 理解意图、评估风险、生成方案 |
| Guard / Policy | 确定性，不可绕过，机器执行 | 「这笔交易符不符合规则」的最终裁决 |
| 链上执行层 | 不可逆，公开，可验证 | 执行并记录 |

Guard 解决的核心风险：**AI 幻觉 × 链上不可逆 = 灾难**。Guard 把「AI 说可以」和「实际执行」之间隔开，确保即使 AI 判断错了，也有机器层面的兜底检查。

## 四、设计原则总结

> **授权不是一次性动作，而是预算 + 范围 + 时间 + 操作类型 + 失败处理的组合。**

> **没有撤销和恢复机制的自动化，不应进入真实资产场景。**

> **规则由人定，执行才交给 Agent——两件事不能颠倒。**
