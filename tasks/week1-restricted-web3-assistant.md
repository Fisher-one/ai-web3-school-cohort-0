# 受限 Web3 助手设计：用稳定币订阅 X Premium

> Week 1 任务｜综合进阶｜设计一个受限 Web3 助手或小 workflow（40 pts）

---

## 一、场景描述

**目标**：用户想用稳定币（USDC）订阅 X Premium，借助 AI 助手完成从信息查询到链上验证的全流程。

**账户类型**：智能账户（Smart Account）+ Session Key

**核心设计思路**：AI 助手负责查询信息和解读结果，不碰私钥；Session Key 在严格限制范围内自动执行支付，超出限制的任何操作都必须人工确认。

---

## 二、前置设置：Session Key 配置（一次性）

在使用助手之前，用户需要完成一次性的 Session Key 配置。这一步本身是链上操作，**必须用户手动签名确认**。

### Session Key 的四个限制维度

| 维度 | 配置值 | 原因 |
|------|--------|------|
| **单笔金额上限** | ≤ 10 USDC | 超出上限自动拒绝，防止大额损失 |
| **每日笔数限制** | ≤ 3 笔/天 | 防止小额多笔被刷走 |
| **收款地址白名单** | 仅限 X 支付处理商地址（用户手动核实填入） | Agent 只能向指定地址转账，防止地址被篡改或注入攻击 |
| **Session Key 有效期** | 30 天自动过期 | 强制定期重新核实地址和权限，避免长期信任过期地址 |

> **关键原则**：白名单地址不能由 Agent 填写。用户必须自己去 X 官网核实支付处理商地址后手动录入——规则由人来定，执行才交给 Agent。

## 三、完整流程

```
[用户] 发起需求
    ↓
[AI] 查询订阅信息
    ↓
[AI] 整理输出确认清单
    ↓
🔴 [用户] 人工确认
    ↓
[AI] 在 Session Key 范围内自动执行支付
    ↓
[AI] 查询链上结果并解读
    ↓
[AI] 告知用户结果
```

### Step 1：用户发起需求

用户用自然语言告诉 AI 助手：

> "帮我用 USDC 订阅 X Premium。"

用户无需提供金额、地址等细节，AI 负责收集。

---

### Step 2：AI 查询订阅信息

AI 助手自动查询以下信息：

- X Premium 订阅套餐与权益（当前为 $8/月）
- 支持的稳定币种类（如 USDC）
- 支付处理商信息（X 使用第三方加密支付处理商，非直接转账）
- 用户当前钱包 USDC 余额

---

### Step 3：AI 整理输出，用户判断是否继续

AI 向用户展示一张确认清单，让用户有足够信息决定「继续 / 不继续」：

```
📋 订阅确认清单
─────────────────────────────
订阅项目：X Premium（基础版）
订阅费用：8 USDC / 月
支付方式：USDC（通过 [支付处理商名称]）
收款地址：0xABC...（已在白名单中）
当前余额：12.5 USDC ✅ 余额充足
─────────────────────────────
你将支付 8 USDC，换取 X Premium 1 个月订阅。
确认继续？
```

---

### Step 4：🔴 用户人工确认

用户核对清单无误后，点击「确认」。

**这一步为什么需要人工？**

Session Key 的白名单和限额是事先设定好的，但「这次要不要买」是用户的主观决策，AI 不应代替用户做消费决定。确认之后，才进入自动执行阶段。

---

### Step 5：AI 在 Session Key 范围内自动执行支付

用户确认后，AI 助手检查本次支付是否满足 Session Key 的所有限制：

- ✅ 金额 8 USDC ≤ 10 USDC 上限
- ✅ 今日已用 0/3 笔
- ✅ 收款地址在白名单内
- ✅ Session Key 未过期

全部通过，AI 自动签名并广播交易，**无需再次弹出 MetaMask 确认窗口**。

> 这正是智能账户比 EOA 体验更好的地方：Session Key 在授权范围内自动执行，省去了每次手动确认的摩擦。

---

### Step 6：AI 查询链上结果并解读

交易广播后，AI 拿到 tx hash，去区块链浏览器查询：

- 交易状态：Success / Failed / Pending
- 实际转账金额：是否与预期一致
- 收款地址：是否为白名单地址
- Gas 费用：实际消耗
- 区块确认数：是否已不可逆

---

### Step 7：AI 告知用户结果

```
✅ 订阅成功
─────────────────────────────
已支付：8 USDC
收款方：[X 支付处理商地址]
交易哈希：0xf76...
区块确认：12 个确认

你的 X Premium 订阅已生效。
```

## 四、边界清单：谁做什么

| 操作 | 谁来做 | 原因 |
|------|--------|------|
| 查询订阅价格和支付信息 | AI | 信息查询，无风险 |
| 核实并填入收款地址白名单 | **用户** | 地址错误无法追回，必须人工核实 |
| 配置 Session Key（一次性） | **用户签名** | 链上操作，必须用户授权 |
| 展示确认清单 | AI | 信息整理，辅助决策 |
| 决定「这次要不要买」 | **用户** | 消费决策不能外包给 AI |
| 在限额内自动执行支付 | AI（Session Key） | 已在授权范围内，无需再次确认 |
| 查询 tx hash 和链上状态 | AI | 数据解读，无风险 |
| 告知结果 | AI | 信息传递 |

---

## 五、风险与限制

| # | 风险 / 限制 | 后果 | 缓解方式 |
|---|------------|------|---------|
| 1 | **白名单地址填错**（用户核实不够仔细，或抄错了地址） | 钱转到错误地址，链上无法追回 | 设置前仔细核对官方渠道；Session Key 有效期到期后强制重新确认 |
| 2 | **支付处理商更新了收款地址**，但白名单还是旧地址 | 交易失败（报错），最多损失一笔 Gas | Session Key 30 天过期机制，强制用户定期重新核实地址 |
| 3 | **AI 从被篡改的页面获取到错误地址**，并展示给用户确认 | 用户误以为地址正确而写入白名单，钱转错 | 白名单地址必须从官方渠道独立核实，不能只依赖 AI 查询结果 |
| 4 | **Session Key 被滥用**（每天 3 笔上限内的小额攻击） | 最多损失 3 × 10 = 30 USDC / 天 | 频率和金额双重限制，定期检查 Session Key 授权记录 |
| 5 | **智能合约代码漏洞**（Session Key 逻辑实现有缺陷） | 限制可能被绕过 | 使用经过审计的合约（如 Safe、Kernel 等），不自己写 Session Key 逻辑 |

---

## 六、一句话总结

> AI 助手负责找信息、执行转账、解读结果；人负责设定规则、核实地址、做消费决策——Agent 在笼子里跑，人来定笼子的边界。
