## 6 个关键标注

| # | 标注项 | 说明 |
|---|--------|------|
| **1** | 谁发起 | **用户** — 自然语言描述需求 |
| **2** | 谁执行 | **Agent (LLM + ReAct)** 负责推理和工具调用；**链上 (EVM)** 负责合约执行 |
| **3** | 签名/付款 | **钱包签名节点** — MetaMask 弹窗确认，用户手动签名；Gas 从用户钱包扣除 |
| **4** | 人工确认 | **两处**：① Agent 生成方案后复核代码/参数 ② 钱包弹出时确认交易内容 |
| **5** | 验证结果 | ① 区块浏览器查 tx 状态 ② 监听 Event 日志 ③ Agent Reflection 自检输出是否对 |
| **6** | 风险点 | 见下方 |

## 主要风险点

| 风险 | 说明 | 影响 |
|------|------|------|
| **Agent 幻觉** | LLM 生成看似正确、实际有 bug 的合约代码 | 部署后无法使用或有漏洞 |
| **RAG 检索不准** | 向量库返回了不相关的 chunk，Agent 基于错误信息做决策 | 方案偏离用户意图 |
| **跳过人工确认** | 用户直接点确认不看内容，或 Agent 未经确认就执行 | 部署了错误合约、转了错误地址 |
| **私钥/授权泄露** | 截图、剪贴板、钓鱼网站获取私钥或签名授权 | 资产全丢 |
| **Gas 预估不准** | Agent 预估 gas 过低，交易失败但仍扣费 | 浪费 ETH |
| **Context window 爆** | RAG 塞太多 chunk + 工具调用历史塞满 context | Agent "忘掉"前面的指令 |

## 这个流程解决什么问题

把 AI Agent 的推理能力和 Web3 链上执行能力串起来，而不是让 Agent 只能"说"不能"做"。核心思路是 **Agent 负责生成方案，人负责关键决策和签名，链负责执行和记录**——三方各管边界内的事。

- AI/Agent 辅助：意图解析、RAG 查资料、合约代码生成、方案构造
- 必须人工确认：代码复核 + 钱包签名（红线：涉及钱和权限的操作）
- 结果验证：tx hash + Event 日志 (链上) + Agent 自检 (链下)
