# EOA、智能账户、多签账户权限对比

> Week 1 任务｜比较 EOA、智能账户和多签账户的权限差异（30 pts）

---

## 一、三类账户对比

| 维度 | EOA | 智能账户（Smart Account） | 多签账户（Multisig） |
|------|-----|--------------------------|---------------------|
| **谁持有控制权** | 单一私钥持有者，私钥 = 全部控制权 | 合约规则持有者，可授权给个人或 Agent | 多个私钥持有者共同持有，需达到门槛（如 3/5）才能操作 |
| **谁可以发起交易** | 私钥持有者本人 | 账户 owner，或在授权范围内的 Agent（Session Key） | 任一成员均可发起提案，但需足够多成员签名通过 |
| **是否支持多人确认** | ❌ 不支持，单一私钥说了算 | ⚠️ 可选，由合约规则决定 | ✅ 核心特性，必须达到 m/n 签名才能执行 |
| **是否支持恢复/限额/自动化** | ❌ 不支持。私钥丢失即永久失去账户 | ✅ 支持社交恢复、限额（Spending Limit）、Session Key 自动化、时间延迟 | ⚠️ 有限支持，可设置门槛，但自动化能力弱 |
| **典型使用场景** | 个人日常使用，少量资产，高频操作 | 个人高价值账户，或需要 Agent 自动化操作的场景 | 团队/DAO 共管金库，多人联合决策 |
| **主要风险点** | 私钥/助记词丢失或泄露，无法恢复，无法撤销 | 合约代码漏洞；规则设置不当（如 Session Key 授权过宽） | 决策效率低，紧急情况协调成本高；签名人私钥丢失过多导致资产永久锁死 |

---

## 二、核心理解

### EOA — 简单但脆弱
私钥是唯一控制手段，没有任何中间层。签名就立刻执行，没有冷静期，没有撤回机制。
- 助记词是私钥的另一种形式，能推导出私钥，同样需要严格保护
- 适合：个人日常小额操作，追求简单直接
- 不适合：大额资产长期存储，或需要自动化的场景

### 智能账户 — 灵活且可编程
账户本身是一个合约，可以在里面写规则。签名和执行被分开了，中间可以插入逻辑：

- **时间延迟**：如 Elytro 的 48h 反劫持窗口——即使攻击者发起了恢复请求，真正的控制权转移要等 48 小时，给 owner 足够时间取消
- **Session Key**：授权 Agent 在特定范围内（限额、限地址、限时间）自动执行操作，不需要每次人工确认——这是它和 EOA 的关键区别
- **Spending Limit**：超出限额的转账需要额外确认

关键认知：**Session Key 的意义就是在授权范围内不需要人工确认**，如果每次都要确认，它和 EOA 没有区别。

### 多签 — 权力分散，适合共同决策
控制权本身就是分散的，没有单一"主人"。适合需要多方制衡的场景（团队金库、DAO 资产管理）。

代价是效率：紧急情况下需要协调多人签名，决策速度慢。另一个风险是签名人密钥管理——3/5 多签中若有 3 人以上丢失私钥，资产永久锁死，门槛永远无法达到。

## 三、哪些操作必须人工确认

| 操作 | 是否需要人工确认 | 原因 |
|------|----------------|------|
| EOA 发起任何转账 | ✅ 必须 | 无任何自动化层，每笔都需签名 |
| 智能账户 Session Key 授权范围内的操作 | ❌ 自动执行 | 这正是 Session Key 的设计目的 |
| 智能账户超出限额的大额转账 | ✅ 必须 | 超出规则边界，需 owner 确认 |
| 多签发起提案 | ✅ 必须 | 提案本身是一次签名操作 |
| 多签成员投票/通过 | ✅ 必须 | 每一票都是独立签名，不能代签 |

---

## 四、一句话总结

> EOA 是「一把钥匙开一把锁」，简单但脆弱；智能账户是「带规则的保险箱」，灵活且可编程；多签是「需要多把钥匙同时开锁」，安全但决策慢。选哪个取决于：控制权该集中还是分散，以及需不需要自动化。
