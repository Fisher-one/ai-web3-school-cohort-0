# AI 基础概念卡片

> 6 个概念，每个用自己的话解释。基于 Handbook 阅读 + 实际使用 Hermes Agent 的经验。

---

## 1. LLM（大语言模型）

**是什么**：一个训练好的神经网络，输入文字，输出文字。本质是概率预测——给你上文，猜下一个 token 最可能是什么。

**我的理解**：LLM 不是数据库，不会"记住"任何东西。它只是参数特别多、见过的文本特别多，所以能根据模式生成看起来合理的回答。但它会瞎编（幻觉），因为它没有"知道真假"的能力——它只是在做概率填空。

**例子**：
- 你问"以太坊是什么"，LLM 根据训练数据里的模式拼出一个回答
- 你让它写 Solidity 合约，它能写——不是因为它"懂"，是因为训练数据里有大量合约代码

**容易踩的坑**：
- 把 LLM 当搜索引擎用——它的知识截止训练日期，之后的事不知道
- 以为它在"推理"——底层是概率匹配，不是逻辑推理
- 问它"你真的理解吗"——它不理解任何东西，只是在预测下一个词

---

## 2. Prompt（提示词）

**是什么**：你发给 LLM 的输入。不只是问题本身，还包括角色设定、格式要求、示例、约束条件。

**我的理解**：Prompt 就是你和模型的沟通协议。写得好，模型输出靠谱；写得烂，模型胡说八道。对比写代码：给函数传参，参数不对输出就错——prompt 也是一样。

**例子**：
- 烂 prompt："解释 Web3" → 模型瞎扯
- 好 prompt："你是 Solidity 开发者，给一个有 3 年后端经验但不懂区块链的人解释什么是智能合约。用类比，控制在 200 字以内。"

**容易踩的坑**：
- 以为 prompt 越长越好——其实精准比长度重要
- 不给角色和约束——模型自由发挥，结果不可控
- prompt 里的模糊词（"一些""大概""稍微"）会让输出随机
- prompt 是有上下文顺序的，靠前的内容权重更高

---

## 3. Context（上下文）

**是什么**：LLM 在当前对话中能"看到"的所有内容——包括你的 prompt、之前的对话、系统指令、RAG 塞进去的文档片段。

**我的理解**：Context 是 LLM 的"临时工作记忆"。没有 context，LLM 就像每次被人叫醒问问题，完全不知道刚才说过什么。Context window（上下文窗口）有大小限制，比如 128K tokens——超了就爆。

**例子**：
- 你问"帮我改这段代码"，LLM 看到的是：系统指令 + 之前的对话 + 你附上的代码 + 你的要求
- 聊天越长，context 越满——所以长对话里 LLM 可能"忘掉"开头说的

**容易踩的坑**：
- Context window 天花板——Agent 每次调工具、读结果都在塞 context，越塞越多
- 不是"给了 context LLM 就懂了"——context 是让 LLM 能参考，不是让它能理解
- context window 不只是大小限制，还有"注意力稀释"问题——内容太多时 LLM 会忽略中间的信息（Lost in the Middle 现象）
---

## 4. RAG（检索增强生成）

**是什么**：让 LLM 能查外部资料再回答。不是把知识硬塞进模型，而是在提问时从向量库里捞相关内容，拼进 prompt 再让 LLM 生成。

**完整链路**：
1. 用户提问
2. Embedding 模型把问题转成向量
3. 向量库做语义检索，拿 top-k 个相关 chunk
4. 可选 rerank 再筛一轮
5. 把 chunk 和原问题拼成 prompt
6. LLM 生成回答
7. 标注 citation

**为什么需要 RAG**：LLM 自己会瞎编，RAG 让它回答有来源可查。而且 LLM 训练完就冻结了，RAG 能查最新信息。

**关键零件**：
| 零件 | 干什么 | 注意事项 |
|------|--------|----------|
| Chunk（切片） | 把文档切成小块 | 切太大语义糊，切太小上下文断。按结构切：title → description → content |
| Embedding 模型 | 文字转向量 | 不同模型算出的向量不能混用 |
| Vector DB | 存向量 | 存的是向量不是原文，搜索靠语义不是关键词 |
| Retriever | 从向量库拿 chunk | 具体交互细节还没搞懂，后续动手补 |
| Rerank | 对结果再排序 | 加了会增延迟，看场景决定用不用 |
| Citation | 标注来源 | 回复里标"来自 Handbook 第三章"，用户能追溯 |

---

## 5. Agent（智能体）

**是什么**：LLM 底座 + 工具调用 + 任务循环。LLM 只管说，Agent 能干活——拆任务、调工具、读结果、再决定下一步。

**核心循环（ReAct）**：
- **Think**：推理下一步该干什么
- **Act**：调工具（查天气、读文件、发 API 请求）
- **Observe**：看工具返回的结果
- 循环，直到任务完成

Planning、Tool Use、Reflection 不是三个独立模块，是同一个循环里的不同环节。

**我的使用体会**（基于 Hermes Agent 日常用）：
- Tool Use 本质就是调 API 再封装——做过后端的不难理解
- Agent 能读能写能改配置能发请求，得画红线：不确定的别编、涉及钱/权限的必须确认、不要替我决定
- Multi-Agent：多个 Agent 各管一块，好处是专注，坏处是出错链式放大

**容易踩的坑**：
- Context window 会爆——每次调工具、读结果都往 context 里塞
- Agent 不是万能的，规划能力有限，复杂任务会跑偏
- 别让 Agent 替你决定需要你确认的事

---

## 6. Embedding（嵌入 / 向量化）

**是什么**：把文字（或图片、音频）转成一串数字（向量）。这个向量代表了文字的语义——意思相近的话，向量在高维空间里离得近。

**我的理解**：Embedding 是 RAG 能工作的基础。没有 embedding，检索只能靠关键词匹配——搜"部署合约"找不到"部署智能合约"。有了 embedding，"听起来像"的就能找到。

**例子**：
- "以太坊是什么"和"介绍一下 ETH" → embedding 相似，能互相检索
- "今天天气"和"智能合约怎么写" → embedding 不相似

**容易踩的坑**：
- 不同 embedding 模型算出来的向量不能混——用 A 模型存的，得用 A 模型搜
- Embedding 是单向的——向量不能还原成原文
