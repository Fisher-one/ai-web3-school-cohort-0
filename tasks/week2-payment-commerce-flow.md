# 最小支付与商业流程拆解

> Week 2 任务｜Payment / Commerce / Settlement（20 pts）

---

## 一、场景描述

**用户对 Agent 说：**
> "查一下过去 7 天 Uniswap 上 USDC/ETH 池的交易量变化，顺便对比 Curve 同池子的数据，给我一份报告。"

Agent 能查公开数据，但发现 Dune Analytics 的历史查询 API 是收费的——单次 $3，批量对比 $5。Agent 需要自己判断「花 $5 拿两份数据值不值」、走 x402 支付、拿到数据、验证数据没被篡改、写成报告交付给你。

**角色分工：**

| 角色 | 是谁 | 做什么 |
|------|------|--------|
| 下单方 | 用户 | 发起需求，设定预算，最终验收报告 |
| 执行方 | Agent | 判断数据来源，发起支付，完成分析，交付报告 |
| 服务提供方 | Dune Analytics | 提供收费数据 API，返回签名数据 |
| 仲裁方 | 预设规则 + 用户 | 数据验证失败或报告质量不达标时介入 |

## 二、完整支付与商业流程

```
用户下单
    ↓
[用户] 设定预算上限（例：本次任务 ≤ 10 USDC）
    ↓
━━━━━━━━━━━━━━━━ Agent 开始执行 ━━━━━━━━━━━━━━━━
    ↓
Agent 查询公开数据
    ├─ 链上数据（Uniswap/Curve 链上事件）：免费，直接读取
    └─ 发现历史聚合数据需要 Dune API → 需要付费
    ↓
【报价阶段】
Agent 评估是否值得付费：
    - Dune 批量查询：$5（在预算 $10 内）
    - 自己爬链上数据：需要额外 2-3 小时计算，成本更高
    - 判断：花 $5 值得，继续
    ↓
【预算授权确认】🔴 人工确认
Agent 向用户展示：
    「需要调用 Dune API，费用 5 USDC。
     在您设定的 10 USDC 预算内。是否授权？」
用户确认 → 生成本次任务的支付授权
    ↓
【执行阶段：x402 支付流程】
Step 1：Agent 请求 Dune API
    → 服务端返回：402 Payment Required
      { "price": "5 USDC", "address": "0xDune...", "deadline": "300s" }

Step 2：Agent 检查付款条件
    - 金额 5U ≤ 预算上限 ✓
    - 收款地址在白名单内 ✓（Dune 官方地址，预先核实）
    - 截止时间 300 秒内可完成 ✓

Step 3：Agent 发起链上支付（在授权范围内自动执行）
    → 5 USDC 转至 Dune 收款地址
    → 拿到 tx hash

Step 4：Agent 带着付款证明重新请求
    GET /api/query?proof={tx_hash}
    → 服务端验证付款 → 返回数据 + 数字签名
    ↓
【数据验证阶段】（自动）
Agent 验证 Dune 返回数据的真实性：
    1. 对返回数据做 hash（keccak256）
    2. 用 Dune 公钥验证数字签名
    3. 签名匹配 → 数据确认未被篡改
    4. 将数据 hash 和 tx hash 一起记录（可供事后审计）

    ⚠️ 注意：签名验证只能证明「数据来自 Dune」，
    不能证明「Dune 的数据是准确的」。
    数据质量判断留给最终验收。
    ↓
【交付阶段】
Agent 整合数据，生成对比分析报告：
    - Uniswap USDC/ETH 7日交易量趋势
    - Curve 同池对比
    - 关键指标差异分析
    - 数据来源 + tx hash + 数据签名（可验证）
    ↓
【验收阶段】🔴 人工确认
用户审阅报告：
    - 数据是否完整？
    - 分析是否合理？
    - 结论是否可信？

    → 接受：流程完成，记录存档
    → 拒绝 / 有争议：进入争议处理
    ↓
【付款 / 退款 / 争议处理】
    ├─ 验收通过 → 本次任务完成，日志归档
    ├─ 数据验证失败（签名不匹配）→ 自动退款请求 + 通知用户
    └─ 报告质量不达标（用户拒绝）→ 人工仲裁：
         · Agent 重新执行（用已付数据）
         · 或用户放弃，损失 5 USDC 数据费
         · （当前场景无 Escrow，数据费已不可逆）
    ↓
【记录与证明】（自动）
写入任务日志：
    - 任务 ID + 时间戳
    - 用户授权记录
    - 支付 tx hash
    - 数据 hash + Dune 签名
    - 报告内容 hash
    - 验收结果
```

## 三、关键设计问题

### 验收的本质问题：自动验证的边界在哪？

数据传输真实性可以自动验证（签名 + hash），但**报告质量无法自动验证**。

「Uniswap 交易量下降 23%」这个结论对不对，需要人来判断。AI 可能选错了时间窗口，可能把手续费计入了交易量，可能忽略了某次异常峰值。这些都不是链上规则能判断的。

**结论：验收必须人工，这是这个场景里不能自动化的终点。**

### 数据签名解决的是什么，解决不了什么

| 问题 | 签名能解决吗 |
|------|------------|
| 数据在传输中被篡改？ | ✅ 能，签名验证覆盖 |
| 数据来源是 Dune 而不是仿冒者？ | ✅ 能，前提是公钥来源可信 |
| Dune 自己的数据是否准确？ | ❌ 不能，这是数据质量问题 |
| Dune 故意返回错误数据？ | ❌ 不能，需要多源交叉验证或 Oracle |

### 当前场景的争议局限

这个场景里数据费（5 USDC）是通过 x402 直接转账给 Dune 的，**没有 Escrow**。一旦付出就无法追回。

如果 Dune 收了钱不返回数据，或返回无效数据，Agent 能做的只有：
1. 记录异常，通知用户
2. 走链下投诉流程（不是链上自动解决）

**这是当前 x402 协议的边界**——它解决了「怎么付款」，没有解决「付了款服务不交付」的问题。更完整的方案需要 Escrow 合约兜底。

## 四、协议对比：x402 vs MPP

### x402：在 HTTP 层解决「怎么付款」

```
客户端请求 → 服务端返回 402 + 付款条件 → 客户端付款 → 带证明重新请求 → 服务端验证 → 返回内容
```

- **解决的问题**：如何在 HTTP 协议里触发和完成一次支付
- **不解决**：服务发现、报价协商、交付验证、争议处理
- **适合场景**：按次付费的 API 调用，小额即时结算，Agent 自主消费

### MPP（Machine Payments Protocol）：机器之间做生意的完整协议

```
服务发现 → 报价（机器可读格式）→ 授权支付 → 执行 → 结算 → 返回收据
```

- **解决的问题**：机器之间的完整商业流程——从发现服务到拿到收据
- **包含 x402 的功能**，但还增加了：报价格式标准、收据格式、错误重试规范
- **适合场景**：Agent 之间的服务交易，需要标准化接口的复杂商业闭环

### 两者的关系

| 维度 | x402 | MPP |
|------|------|-----|
| 定位 | HTTP 支付补丁 | 完整机器商业协议 |
| 解决的链路段 | 付款触发 + 付款证明 | 发现 → 报价 → 支付 → 收据 |
| 复杂度 | 低，易接入 | 高，需要双方都支持协议 |
| 适合阶段 | 早期快速验证 | 需要标准化的生产场景 |

**在本场景中**：Dune Analytics 实现 x402 就够了——简单的按次付费 API，不需要 MPP 的完整协议栈。如果是 Agent 之间相互调用服务（比如 Agent A 雇用 Agent B 做数据分析），MPP 会更合适，因为双方都需要机器可读的报价、收据和错误处理标准。

### ERC-8004 和 ERC-8183：补全剩余两段链路

**ERC-8004：解决「谁在付」**

x402 和 MPP 解决的是「怎么付」，但没有定义付款方是谁。当 Agent 代表用户发起支付时，服务方需要知道：这个 Agent 是谁授权的？它有没有资格发起这笔交易？

ERC-8004 是 Agent 的身份声明标准——链上可验证的「我是谁、我被谁授权、我能做什么」。在本场景中，Dune 收到付款后，可以通过 ERC-8004 验证「这个 Agent 确实被该用户授权来购买数据」，而不只是收到了一笔来路不明的转账。

**ERC-8183：解决「收据长什么样」**

MPP 协议规定了「结算后必须返回收据」，但没有定义收据的具体格式。ERC-8183 是链上收据的格式标准——定义了收据包含哪些字段（付款方、收款方、金额、服务描述、时间戳、数据 hash 等），让机器可以直接解析，不需要每家服务商自定义格式。

在本场景中，Dune 返回的收据如果符合 ERC-8183，Agent 就可以自动解析并存入日志，无需人工处理格式差异。

**四个协议的链路位置：**

| 协议 | 解决链路中的哪一段 |
|------|-----------------|
| x402 | 付款触发 + 付款证明（HTTP 层） |
| MPP | 完整商业流程：发现 → 报价 → 支付 → 收据 |
| ERC-8004 | 付款方身份：这个 Agent 是谁、被谁授权 |
| ERC-8183 | 收据格式：结算凭证长什么样、机器怎么读 |

---

## 五、一句话总结

> Payment 只是链路中的一段。x402 解决了怎么付款，但「值不值得付、数据可不可信、报告质量够不够」——这三个问题分别由预算策略、签名验证和人工验收来回答，缺一不可。
